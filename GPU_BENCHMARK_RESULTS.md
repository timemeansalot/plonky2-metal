# Lighter-Prover GPU Acceleration Benchmark Results

**Date:** 2026-03-10 (updated from 2026-03-09)
**Hardware:** Apple M4 (MacBook Pro, 10-core CPU, 10-core GPU)
**Benchmark:** 500 transactions, 125 chunks of 4 txs each

## Summary

Numbers are from full 125-chunk cold-start runs (machine rebooted before each benchmark). Actual performance may vary with machine temperature and workload.

| Configuration | Total Proving Time | Speedup vs CPU |
|--------------|-------------------|----------------|
| CPU-only | 602s | baseline |
| GPU Metal (Merkle + Quotient) | 374-385s | **1.56-1.61x (36-38% faster)** |

Two independent runs confirm the results:

| Run | Commit | BlockTx avg | BlockTxChain avg | Total | Speedup |
|-----|--------|-------------|-----------------|-------|---------|
| Run 1 (cold) | `5ef45654` | 2.47s | 518ms | ~374s | 1.61x |
| Run 2 (cold) | `81113964` | 2.55s | 525ms | ~385s | 1.56x |
| CPU baseline (cold) | upstream `e1c2d35` | 4.08s | 730ms | ~602s | — |

With pipeline overlap (BlockTxChain runs concurrently with next BlockTx on a separate thread), the wall-clock time improves further by hiding chain proving time behind the larger BlockTx proof.

## GPU Acceleration

Two Metal GPU kernels are used:

1. **Merkle tree hashing** (Poseidon2) — GPU replaces CPU for trees with 8K+ leaves.
2. **Quotient polynomial evaluation** — fused gate evaluation + alpha reduction, zero device memory allocation. This is the main performance win.

All proofs generated by the GPU path are **correct and verified** — the full 125-chunk benchmark completes without errors, and BlockTxChainCircuit recursively verifies each BlockTx proof inside its circuit, confirming validity.

## Circuit Structure

The benchmark runs: **1 PreExec + 125 iterations of (BlockTx + BlockTxChain)**.

| Circuit | Inner Sub-proofs | Sizes |
|---------|-----------------|-------|
| BlockPreExecutionCircuit | 2 sub-proofs | 2 x small (degree 2^14, 16 gate types) |
| BlockTxCircuit | 2 sub-proofs | 1 x small (degree 2^14, 16 gate types) + **1 x large (degree 2^16, 27 gate types)** |
| BlockTxChainCircuit | 1 sub-proof | 1 x small (degree 2^14, 16 gate types) |

BlockTxCircuit dominates total proving time (85%) due to its large inner sub-proof (degree 2^16, 4x the evaluation domain size of small sub-proofs).

## Key Bug Fix: BaseSumGate Parameter Parsing

During GPU quotient development, a parsing bug caused the large sub-proof (degree 2^16) to produce wrong constraint values. The GPU kernel parses gate parameters from ID strings like:

```
"BaseSumGate { num_limbs: 16 } + Base: 4"
```

The parser searched for `"Base"` but matched it at position 0 in `"BaseSumGate"` instead of in `"+ Base: 4"`, causing the base parameter to silently default to 2 instead of the correct value 4. This has been fixed by searching for `"+ Base"` instead of `"Base"`.

## How to Integrate

The GPU acceleration is **fully transparent** — no changes to circuit code or proving logic are needed. The GPU kernels are automatically dispatched at runtime when the conditions are met (GoldilocksField + Poseidon2Hash + Apple Silicon).

### Requirements

- **macOS** with **Apple Silicon** (M1/M2/M3/M4)
- Metal GPU framework (included in macOS by default)

### Step 1: Use the Metal-enabled plonky2 fork

In your `Cargo.toml`, add `features = ["metal"]` to the plonky2 dependency:

```toml
# GPU-accelerated:
plonky2 = { git = "https://github.com/timemeansalot/plonky2-metal", package = "plonky2", features = ["metal"] }
```

That's it. No other code changes are required.

### What happens automatically

When `features = ["metal"]` is enabled, two proving steps are offloaded to the GPU at runtime:

1. **Merkle tree construction** (`MerkleTree::new`) — GPU Poseidon2 hashing for trees with 2^13 to 2^20 leaves
2. **Quotient polynomial evaluation** (`compute_quotient_polys`) — GPU fused gate evaluation + alpha reduction

Both are gated by runtime `TypeId` checks:
- Field type must be `GoldilocksField`
- Hasher must be `Poseidon2Hash` (for Merkle)
- No lookup gates (for quotient poly)

If any condition is not met, the code falls back to the original CPU path transparently. On non-macOS platforms, the `metal` feature simply has no effect — the `#[cfg(feature = "metal")]` blocks are compiled out.

### Step 2 (Optional): Pipeline overlap

For workloads that prove multiple chunks sequentially (like the bench), you can overlap independent proofs on separate threads. For example, if circuit B depends on the proof of circuit A but not on circuit C, run circuit C concurrently with circuit B:

```rust
// Main thread: prove circuit A sequentially
let proof_a = CircuitA::prove(...);

// Send proof_a to worker thread for circuit C (runs concurrently)
chain_sender.send(proof_a.clone()).unwrap();

// Main thread: immediately start next circuit A (no wait for C)
let proof_a_next = CircuitA::prove(...);
```

This is application-level and depends on your specific proof dependency graph. In the bench, this hides BlockTxChain time behind BlockTx, saving an additional wall-clock time.

### No shader compilation needed

The Metal shader libraries (`.metallib`) are **pre-compiled and embedded** into the plonky2 binary via `include_bytes!` at build time. There is no need to install Metal tooling or compile shaders separately.
